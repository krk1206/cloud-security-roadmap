## 오늘 공부한 내용
### 1. 클라이언트-서버 구조 (Web)
- **Request & Response**: 사용자가 브라우저(클라이언트)로 주소(`google.com/index.html`)를 요청하면, 서버는 저장소에서 해당 파일(`index.html`)을 찾아 응답함.
- **Localhost**: 자기 자신의 웹 페이지에 접속할 때도 동일한 메커니즘으로 동작.
- **설정 파일**: 리눅스 시스템의 설정은 주로 `/etc` 디렉토리에 저장됨.

### 2. 로그 관리 (Monitoring)
- **실시간 로그 확인**: `tail -f` 명령어를 사용하여 서버에 누가 접속하는지 실시간으로 모니터링 가능.
  - 예: `tail -f /var/log/apache2/access.log`
  - 로그는 서버 관리 및 에러 추적의 핵심 자산임.

### 3. SSH (Secure Shell)
- **원격 제어**: 클라이언트(내 PC)에서 서버(원격지)로 접속하여, 마치 서버 앞에서 작업하는 것처럼 명령어를 실행할 수 있음.
- **Server & Client**: 접속을 받는 쪽은 `ssh server`, 접속하는 쪽은 `ssh client`가 설치되어 있어야 함.

### 4. 포트(Port)와 보안
- **포트의 개념**: 서버 내의 특정 서비스로 들어가기 위한 '문(Door)' 또는 '구멍'.
- **보안 설정**: SSH의 기본 포트(22번)는 공격의 대상이 되기 쉬우므로, 설정 파일(`sshd_config`)에서 포트 번호를 변경하는 것이 보안상 권장됨.

## ✅ 이해한 것
- **웹의 기본 동작**: "요청(Request) → 파일 찾기 → 응답(Response)"의 흐름을 이해함.
- **로그의 중요성**: `access.log` 등을 통해 누가 언제 접속했는지 감시할 수 있으며, 이는 모든 서버 운영의 공통점임.
- **SSH의 위험성**: 원격에서 모든 제어가 가능하다는 편리함 뒤에, 탈취될 경우 치명적일 수 있다는 보안 위협이 있음을 인지함. 포트 변경 등의 보안 조치가 필요함.

## 🤔 헷갈린 것 & 해결
### 1. HTTP 상태 코드 (Status Code)
- **문제**: 로그 파일에 찍히는 200, 404, 500 같은 숫자의 의미가 헷갈림.
- **해결**: 백의 자리 숫자로 큰 범주를 먼저 외우기.
  - **2xx (성공)**: "잘 처리됨" (예: **200 OK**)
  - **3xx (리다이렉션)**: "다른 주소로 가세요" (이사 감)
  - **4xx (클라이언트 에러)**: "니가(사용자) 실수함" (예: **404 Not Found** - 파일 없음, 주소 오타)
  - **5xx (서버 에러)**: "내가(서버) 실수함" (예: **500 Internal Server Error** - 서버 설정 오류나 코드 문제)
